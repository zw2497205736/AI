# Agent 工具架构治理

这个 skill 只服务一个明确目标：

- **让 ChatAgent 的工具体系保持清晰、可扩展、可维护**

不要把它用于普通问答、前端展示、部署问题、README 编写。

## 适用场景

只有在这些场景才使用：

1. 新增一个 ChatAgent 工具
2. 修改已有工具的输入输出
3. 判断某个能力应不应该做成工具
4. 判断某个能力应不应该走 MCP
5. 重构 ChatAgent 的工具路由逻辑

## 本项目里的真实痛点

随着聊天模块从普通问答演进到 ChatAgent，工具系统会越来越容易出现这些问题：

1. **工具命名越来越乱**
   同一类能力命名风格不统一，后面自己都难维护。

2. **工具边界不清**
   本来应该拆成两个小工具，结果糊成一个大工具；或者本来应该是普通 RAG，结果硬做成工具。

3. **路由规则越来越复杂**
   工具一多，规则匹配、模型选择、回退逻辑容易缠在一起。

4. **内置工具和外部工具职责混杂**
   平台内部数据、知识库、GitHub 实时数据、本地状态，全都混在一起，会让架构越长越乱。

这个 skill 的目标就是给工具体系建立清晰边界。

## 设计原则

### 1. 一个工具只做一件事

工具应该是细粒度的。

正确做法：

- `list_connected_repositories`
- `list_recent_github_tasks`
- `get_github_task_detail`
- `search_knowledge_base`

不推荐这种大而乱的设计：

- `github_tool`
- `platform_query_tool`
- `knowledge_tool`

### 2. 工具必须有清晰输入输出

每个工具至少要明确：

- 解决什么问题
- 输入参数是什么
- 返回结果是什么
- 失败时返回什么

如果这些说不清，就说明工具边界没设计好。

### 3. 优先保留内置工具，不要为了 MCP 而 MCP

本项目里有两类能力：

#### 平台内部能力

例如：

- 已接入仓库
- 任务列表
- 任务详情
- 文档列表
- 知识库搜索

这类优先做成内置工具。

#### 外部实时能力

例如：

- GitHub 实时 PR 信息
- GitHub PR 文件变更
- 外部平台实时状态

这类才考虑走 MCP 或外部工具。

原则：

- 平台内部数据优先内置
- 外部实时数据再考虑 MCP

### 4. 工具名要统一

本项目工具命名建议统一为：

- 动词 + 领域对象

例如：

- `list_connected_repositories`
- `list_recent_github_tasks`
- `get_github_task_detail`
- `search_knowledge_base`
- `github_get_pull_request`

不要出现一会儿中文拼音、一会儿驼峰、一会儿大而泛的命名。

### 5. 路由逻辑要分层

工具路由不要一股脑堆在一起。

建议按这 3 层理解：

1. **规则优先路由**
   对明显问题直接命中工具

2. **模型辅助决策**
   对模糊问题由模型判断是否调用工具

3. **回退逻辑**
   工具未命中时退回 RAG 或普通回答

如果三层揉在一起写，后面一定越来越乱。

## 工具分类建议

本项目建议把工具分成 3 类：

### 1. 平台内置工具

查询平台数据库和系统状态：

- 仓库
- 任务
- 文档
- 会话
- 记忆

### 2. 知识库工具

围绕 RAG 和知识库检索：

- 搜索知识库
- 返回知识片段
- 返回来源文件

### 3. 外部协作工具

面向 GitHub 这类外部系统：

- PR 查询
- PR 文件查询
- issue 查询
- commit 查询

## 判断某个能力要不要做成工具

新增能力前，先问这 4 个问题：

1. 这个能力是不是一个独立动作？
2. 这个动作有没有明确输入输出？
3. 这个动作是不是用户会反复问？
4. 这个动作是不是比直接塞进 Prompt 更清晰？

如果 4 个问题里至少 3 个是“是”，就适合做成工具。

## 判断某个能力要不要走 MCP

只有满足下面条件时，才优先考虑 MCP：

1. 数据来自外部系统
2. 需要实时查询
3. 平台本地并没有完整缓存
4. 未来可能继续扩展更多外部工具

如果不满足这些条件，就不要为了“看起来高级”硬走 MCP。

## 新增工具时的检查清单

每次新增或修改工具后，至少检查：

1. 工具名是否统一
2. 工具边界是否足够小
3. 参数是否清楚
4. 返回结果是否稳定
5. 属于内置工具还是外部工具是否判断正确
6. 路由规则是否会和现有工具冲突

## 最终标准

这个 skill 的目标不是让工具越来越多，而是让工具体系越来越清晰。

一个合格的工具系统应该满足：

1. 新工具一看名字就知道干什么
2. 工具职责不混乱
3. 工具路由不容易失控
4. 后续继续加工具时，不会把 ChatAgent 改成一锅粥
