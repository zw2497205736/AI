# PR 审查治理

这个 skill 只服务一个明确目标：

- **让 GitHub PR 自动审查输出更稳定、更统一、更像可复用的研发规范**

不要把它用于普通聊天、知识库问答、部署问题、前端样式修改。

## 适用场景

只有在这些场景才使用：

1. 修改 GitHub PR 自动审查相关 Prompt
2. 修改 PR 自动审查结果的输出结构
3. 评审某次 Code Review / 测试建议输出是否太飘
4. 给 PR 自动审查建立统一标准

## 本项目里的真实痛点

GitHub PR 自动审查虽然已经能跑，但长期使用最容易出现这些问题：

1. **不同 PR 输出粒度不一致**
   有的很细，有的很空。

2. **风险点覆盖不稳定**
   有时能抓到逻辑问题，有时只输出很表面的建议。

3. **测试建议容易泛化**
   常常变成“建议增加边界测试”这种空话。

4. **单元测试建议不够贴代码**
   输出看起来像测试，但不一定真贴当前改动。

这个 skill 的目标就是围绕这些问题，给 PR 自动审查建立统一的审查口径。

## 管理范围

重点关注这些位置：

- `backend/prompts/github_pr_review.py`
- `backend/prompts/github_pr_testing.py`
- `backend/services/agent_service.py`
- `backend/routers/github_router.py`
- GitHub Agent 页对应的前端展示逻辑

## 审查统一标准

对每个 PR，优先统一覆盖这 4 类内容：

1. **逻辑问题**
2. **边界条件**
3. **回归风险**
4. **测试缺口**

如果输出没有覆盖到这 4 类中的主要部分，就说明审查标准不够稳。

## Code Review 输出要求

Code Review 的重点必须优先放在：

- 逻辑正确性
- 空值 / 边界处理
- 兼容性与回归风险
- 可维护性问题

不要把重点放在：

- 命名吹毛求疵
- 格式细节
- 没有工程风险的风格建议

### 合格输出应该像这样

- 能指出具体问题点
- 能说明为什么有风险
- 能说明可能造成什么后果
- 最好能给出修改方向

### 不合格输出常见表现

- “建议优化代码结构”
- “建议增加可读性”
- “建议补充边界测试”

这类话如果没有结合具体改动点，就是无效输出。

## 测试建议输出要求

测试建议必须围绕当前 PR 改动本身展开。

优先覆盖：

- 主路径
- 边界输入
- 空值 / 异常值
- 回归风险点
- 输入格式差异

测试建议不应该只停留在抽象层面，而应该尽量给出：

- 测试对象
- 建议输入
- 预期行为

## 单元测试建议输出要求

单元测试建议应该尽量贴近“可写成测试代码”的粒度。

至少要回答清楚：

- 测什么方法 / 函数 / 模块
- 输入是什么
- 断言什么

如果只是说“建议写单测覆盖异常路径”，那不够。

## 修改 PR 审查 Prompt 时的规则

### 1. 先看当前 PR 数据输入长什么样

改 Prompt 前，先确认系统实际给模型提供了什么：

- 是完整 diff 还是截断 diff
- 有没有文件名、patch、增删行数
- 有没有 PR 标题和事件类型

如果上下文里没有，就不要在 Prompt 里假设模型已经知道。

### 2. Prompt 要优先引导“发现具体问题”

不要让模型默认输出一堆空泛建议。

Prompt 里应该显式引导：

- 找真实风险
- 找边界问题
- 找测试缺口
- 找会影响功能正确性的点

### 3. 审查结果结构要稳定

同类型输出尽量使用固定结构，例如：

- 问题
- 风险
- 建议

或者：

- 测试重点
- 建议用例
- 回归点

不要每次一换 Prompt，结果结构就完全变。

## 自动检查清单

每次调整 PR 自动审查相关逻辑后，至少检查：

1. 输出有没有覆盖逻辑 / 边界 / 风险 / 测试
2. 输出是不是还在说空话
3. 测试建议是不是足够具体
4. 单测建议是不是贴当前代码改动
5. 前端展示是否还能清晰承接结果

## 最终标准

这个 skill 的目标不是“让 AI 说得更多”，而是：

1. **说得更具体**
2. **说得更稳定**
3. **更贴当前 PR 改动**
4. **更像团队内可复用的审查规范**

如果做不到这几点，就说明这次 PR 自动审查治理还不够到位。
